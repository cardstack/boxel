# ImageDef cheatsheet

The canonical reference for stored / uploaded / linked image fields in Boxel. For external URL-only images and the URL/ImageDef pair pattern, see [`base-field-catalog.md`](./base-field-catalog.md) "Image fields — the URL/ImageDef pair pattern."

## What ImageDef is

`ImageDef extends FileDef` — defined in `https://cardstack.com/base/card-api` (line 2854 of the source). It's the file-backed image type: an `ImageDef` instance is a real realm file with content-type `image/*`, a URL, and metadata. `linksTo(ImageDef)` says "this field holds a reference to that realm file."

`ImageCard` is a separate, deprecated class — defined in `https://cardstack.com/base/image`. It's a URL-holding CardDef, not file-backed. **Do not use `ImageCard` for new schemas.** The two are commonly confused because they both render images.

## Import — three equivalent valid forms

```ts
// Recommended for new code — explicit about which class you want
import ImageDef from 'https://cardstack.com/base/image-file-def';

// Equivalent — named import from the source module
import { ImageDef } from 'https://cardstack.com/base/card-api';
```

```ts
// FORBIDDEN — this gets ImageCard (deprecated), not ImageDef
import ImageDef from 'https://cardstack.com/base/image';   // ← WRONG
```

The local alias `ImageDef` makes lint happy on the wrong form. At runtime you get ImageCard, the field schema mismatches what your template expects, and downstream loads fail. Verify the right module by probing:

```sh
npx boxel run-command @cardstack/boxel-host/tools/get-card-type-schema/default \
  --realm "$REALM" \
  --input '{"codeRef":{"module":"https://cardstack.com/base/image-file-def","name":"default"}}' \
  --json
# status: ready → the file-backed ImageDef exists at that URL
```

## Schema usage

```ts
// Single linked image
@field heroImage = linksTo(() => ImageDef);

// Many linked images (gallery)
@field galleryImages = linksToMany(() => ImageDef);
```

Use the `() => ImageDef` thunk form by default — defensive against any future cycle. The base-class case is technically safe without a thunk, but uniform thunk-by-default is the discipline that prevents bugs.

**Never `contains(ImageDef)`** — ImageDef is a CardDef-shaped thing (extends FileDef), referenced via link, not embedded as a value.

## Instance JSON

Point to the actual realm file with its extension:

```json
"relationships": {
  "heroImage": {
    "links": { "self": "../GeneratedImages/NBJ/beaumont-hero.jpg" }
  }
}
```

The path must start with `./`, `../`, or be a full URL — see [`core-concept.md`](./core-concept.md) "Relationship path resolution." Bare paths fail.

**Never put an external image URL in `links.self`** — that bricks the realm via Postgres NULL-byte rejection on the indexer's failed JSON parse of the binary response. External URLs go in the paired `*URL` field (UrlField-typed) in attributes. See `base-field-catalog.md` for the pair pattern.

## Rendering

```hbs
{{!-- Built-in FileDef rendering (handles loading, alt, crop) --}}
<@fields.heroImage @format='fitted' />

{{!-- Template-owned rendering (when you need pixel control) --}}
{{#if @model.heroImageURL}}
  <img src={{@model.heroImageURL}} alt={{@model.title}} class='hero' />
{{else if @model.heroImage}}
  <img src={{@model.heroImage.url}} alt={{@model.title}} class='hero' />
{{/if}}
```

The `.url` property on an ImageDef instance gives you the raw file URL. Combine with the URL-twin pattern to prefer external URLs and fall back to linked realm files.

## Generated / uploaded bytes

```ts
// Generated by an AI image model, returned as data URI or remote URL
import UploadImageCommand from '@cardstack/catalog/commands/upload-image';

const upload = new UploadImageCommand(this.commandContext);
const result = await upload.execute({
  sourceImageUrl: base64DataUri,  // or remote URL
  targetRealmUrl: realm,
});
// result.cardId is the new file's card URL — assign to your linksTo field
```

`UploadImageCommand` handles base64 / blob / remote URLs and writes the actual file plus a wrapping card. For non-image generic binaries, use `WriteBinaryFileCommand`.

**Never inline image bytes in card JSON.** No `data:`, `blob:`, base64 in StringField, attributes, or notes. Always write the bytes as a realm file first, then `linksTo(ImageDef)` to reference it.

## Decision table

| You have | Field shape | Where it goes |
|---|---|---|
| External URL (Unsplash, CDN, S3) | `contains(UrlField)` | `attributes.<field>URL` as a string |
| Generated image bytes (from AI model) | `linksTo(() => ImageDef)` after `UploadImageCommand` | `relationships.<field>.links.self` as a realm file path |
| User-uploaded image | `linksTo(() => ImageDef)` | Same — uploaded via host UI or `WriteBinaryFileCommand` |
| Gallery of any of the above | `linksToMany(() => ImageDef)` + `containsMany(UrlField)` twin | Indexed-key relationships + array of URLs |
